🍔 MENU - Menús del Sistema

Guía de Desarrollo de User Stories

📋 Descripción General

Propósito: Módulo de navegación principal del sistema que gestiona menús basados en roles (back-office y admin), permitiendo acceso a funcionalidades mediante selección numérica o click directo.

Contexto de Negocio: El módulo MENU es la puerta de entrada principal a todas las funcionalidades del sistema CardDemo. Permite a los usuarios navegar de forma intuitiva según sus permisos, emulando la experiencia de sistemas mainframe tradicionales pero con una interfaz moderna y responsive.

Roles Soportados:

  • Back-office: Acceso al menú principal con funciones operativas
  • Admin: Acceso a menú principal y menú administrativo

📝 Templates Específicos de User Stories

US-MENU-001: Visualización de Menú Principal

Como usuario autenticado del sistema, quiero visualizar el menú principal con todas las opciones disponibles según mi rol, para poder acceder rápidamente a las funcionalidades que necesito.

Criterios de Aceptación:
  • El menú muestra título "CardDemo - Main Menu" y subtítulo
  • Se muestran todas las opciones numeradas del 1 al 10
  • Cada opción incluye label y descripción
  • El diseño es responsive (2 columnas en pantallas anchas)
  • Las opciones solo-admin muestran badge "Admin"
  • El header muestra transactionId "CC00" y programName "COMEN01"
US-MENU-002: Selección de Opción por Número

Como usuario del menú, quiero ingresar un número (01-10) y presionar Enter para navegar a la funcionalidad seleccionada, facilitando la navegación rápida sin usar el mouse.

Criterios de Aceptación:
  • Input acepta solo números de máximo 2 dígitos
  • Botón "Continue" se habilita solo con input válido
  • Al presionar Enter o click en Continue, se navega a la ruta de la opción
  • Se muestra loading state durante la navegación
  • Opciones deshabilitadas no se pueden seleccionar
  • Se muestra error si la selección es inválida
US-MENU-003: Selección de Opción por Click

Como usuario del menú, quiero hacer click directamente en cualquier opción para navegar a esa funcionalidad, proporcionando una alternativa moderna a la selección numérica.

Criterios de Aceptación:
  • Cada opción es clickeable (ListItemButton)
  • Hover muestra traslación (8px) y sombra
  • Click en opción habilitada navega a su ruta
  • Opciones disabled no responden a clicks
  • Badge numerado cambia a color primario en hover
US-MENU-004: Acceso a Menú Administrativo

Como usuario con rol admin, quiero acceder al menú de administración con funciones de seguridad y gestión de usuarios, para realizar tareas administrativas del sistema.

Criterios de Aceptación:
  • Solo usuarios admin pueden acceder a /menu/admin
  • El menú muestra título "CardDemo - Administration Menu"
  • Se muestran 4 opciones de seguridad (User List, Add, Update, Delete)
  • transactionId es "CADM" y programName "COADM01"
  • Usuarios back-office son redirigidos a /menu/main
  • Icono AdminPanelSettings se muestra en el header
US-MENU-005: Salida del Menú (F3/Exit)

Como usuario del menú, quiero presionar F3 o hacer click en el botón Exit para cerrar sesión y regresar al login, asegurando que puedo salir del sistema de forma segura.

Criterios de Aceptación:
  • Presionar F3 ejecuta logout
  • Presionar ESC ejecuta logout
  • Click en botón "F3 = Exit" ejecuta logout
  • Se ejecuta dispatch(logoutUser()) de Redux
  • Usuario es redirigido a /login
  • Estado de autenticación se limpia completamente
US-MENU-006: Diseño Responsive del Menú

Como usuario móvil, quiero que el menú se adapte al tamaño de mi pantalla, para poder navegar cómodamente desde cualquier dispositivo.

Criterios de Aceptación:
  • Pantallas ≥1360px: Grid de 2 columnas
  • Pantallas móviles: Grid de 1 columna
  • Pantallas <720px: Altura de lista ajustada dinámicamente
  • Scroll vertical funciona cuando opciones exceden altura
  • Botones y campos mantienen tamaño legible en móvil
  • Layout usa padding responsive (xs: 2, sm: 3)

⚡ Factores de Aceleración del Desarrollo

🔄 Componente MenuScreen Reutilizable

Aceleración: ~70% de reducción en tiempo de desarrollo

Cómo usarlo: MenuScreen es un componente genérico que acepta MenuData como prop. No es necesario crear nuevos componentes de menú, solo definir los datos.

<MenuScreen menuData={getCustomMenuData()} onOptionSelect={handleOptionSelect} onExit={handleExit} error={error} loading={loading} />

🎣 Hook useMenu

Aceleración: ~50% de reducción en lógica de negocio

Cómo usarlo: Proporciona toda la lógica de navegación, manejo de errores, loading states, y logout automáticamente.

const { loading, error, handleOptionSelect, handleExit } = useMenu({ onError: (error) => console.error('Error:', error), onSuccess: (option) => console.log('Selected:', option.label), });

📊 Estructura de Datos menuData.ts

Aceleración: ~30% de reducción en configuración

Cómo usarlo: Simplemente agregar funciones getters que retornen MenuData. No se requiere configuración de rutas adicionales.

export const getMyMenuData = (): MenuData => ({ title: 'Mi Menú', subtitle: 'Funciones Personalizadas', transactionId: 'CUST', programName: 'CUSTOM01', userRole: 'back-office', options: [ { id: 'opt1', label: 'Opción 1', path: '/custom/1' }, { id: 'opt2', label: 'Opción 2', path: '/custom/2', adminOnly: true }, ], });

🎨 Material-UI Components

Aceleración: ~80% de reducción en desarrollo de UI

Disponibles: Container, Paper, List, ListItemButton, TextField, Button, Chip, Alert, todos pre-configurados con estilos consistentes del sistema.

📊 Guías de Complejidad

Simple 1-2 puntos

  • Agregar nueva opción a menú existente: Solo modificar array de options en menuData.ts
  • Cambiar texto/descripción de opción: Actualizar label o description
  • Deshabilitar opción temporalmente: Agregar disabled: true
  • Cambiar rol requerido de opción: Modificar requiredRole o adminOnly

Medio 3-5 puntos

  • Crear nuevo menú completo: Nueva función en menuData.ts + nueva página similar a MainMenuPage
  • Agregar validación custom antes de navegar: Extender hook useMenu con lógica adicional
  • Implementar opción con acción (no path): Usar campo action y manejar en useMenu
  • Agregar subniveles al menú: Modificar estructura de MenuOption para soportar children
  • Personalizar estilos del menú: Extender MenuScreen con props de theme customización

Complejo 5-8 puntos

  • Implementar menú dinámico desde API: Fetch de opciones desde backend + actualización de types
  • Agregar búsqueda/filtrado de opciones: Nuevo campo de búsqueda + lógica de filtrado en MenuScreen
  • Implementar favoritos/recientes: Almacenamiento local + reordenamiento de opciones
  • Soporte multi-idioma en menús: Integración con i18n + traducción de labels/descriptions
  • Analytics de uso de opciones: Tracking de clicks + envío a analytics + dashboard
  • Permisos granulares por opción: Sistema de permisos más complejo que solo admin/back-office
💡 Tip de Estimación: Las tareas simples se resuelven modificando solo datos (menuData.ts). Las tareas medias requieren lógica nueva (hooks/validaciones). Las tareas complejas implican cambios arquitectónicos (nuevos módulos/integraciones).

🔧 Fundamentos Técnicos

🏗️ Componentes Clave

Componente Ubicación Responsabilidad
MenuScreen /app/components/menu/MenuScreen.tsx Componente de UI reutilizable para renderizar cualquier menú
useMenu /app/hooks/useMenu.ts Hook con lógica de navegación, loading, y manejo de errores

menu types /app/types/menu.ts Interfaces TypeScript (MenuData, MenuOption)

🔗 Dependencias Principales

Módulos Internos

  • AUTH: Verificación de autenticación y control de acceso por roles
  • Layout: GlobalHeader component para encabezado consistente
  • Store: Redux hooks (useAppSelector, useAppDispatch) y authSlice

Bibliotecas Externas

  • React Router: Navegación (useNavigate, rutas)
  • Material-UI: Componentes de UI (Container, Paper, List, TextField, Button, etc.)
  • Redux Toolkit: Gestión de estado global (autenticación)

🌐 Interfaces Públicas (Mock API)

GET /api/menu/main Respuesta: { success: true, data: MenuData } Descripción: Obtiene datos del menú principal GET /api/menu/admin Respuesta: { success: true, data: MenuData } Descripción: Obtiene datos del menú administrativo POST /api/menu/validate Request: { optionId: string, menuType: 'main' | 'admin' } Respuesta: { success: true, data: { validated: true, redirectUrl: string } } Descripción: Valida selección de opción

📊 Modelo de Datos

interface MenuData { title: string; // Título del menú subtitle?: string; // Subtítulo opcional transactionId: string; // ID transacción (legacy COBOL) programName: string; // Nombre programa (legacy COBOL) userRole: 'admin' | 'back-office'; // Rol del usuario options: MenuOption[]; // Array de opciones } interface MenuOption { id: string; // ID único label: string; // Texto visible description?: string; // Descripción path?: string; // Ruta de navegación action?: string; // Acción personalizada disabled?: boolean; // Si está deshabilitada requiredRole?: 'admin' | 'back-office' | 'both'; adminOnly?: boolean; // Solo para admin }

🎯 Reglas de Negocio Clave

⚠️ Control de Acceso:
  • Menú Principal: Accesible por 'admin' y 'back-office'
  • Menú Admin: Solo accesible por 'admin'
  • Usuarios no autenticados → Redirigir a /login
  • Back-office intentando acceder a admin → Redirigir a /menu/main
📋 Validaciones de Selección:
  • Input acepta solo números (máximo 2 dígitos)
  • Opción debe existir en array de options
  • Opción no debe estar marcada como disabled
  • Botón Continue deshabilitado si input vacío
✅ Comportamiento Responsive:
  • ≥1360px: Grid de 2 columnas, max height 80vh
  • <720px: Altura calculada dinámicamente (viewport - header - footer - 40px)
  • Scroll personalizado cuando opciones exceden altura disponible
  • Header y footer medidos con ResizeObserver

💡 Ejemplos de User Stories Completas

Ejemplo 1: Agregar Nueva Opción al Menú Principal

Historia:

Como Product Owner, quiero agregar la opción "Customer Reports" al menú principal, para que los usuarios back-office puedan acceder a reportes de clientes.

Tareas Técnicas:
  1. Modificar /app/data/menuData.ts → función getMainMenuData()
  2. Agregar objeto al array options:
    { id: 'customer-reports', label: 'Customer Reports', description: 'Generate customer activity reports', path: '/reports/customers', }
  3. Crear página /app/pages/reports/CustomerReportsPage.tsx
  4. Agregar ruta en React Router
Estimación: 2 puntos

Ejemplo 2: Implementar Menú de Reportes Dinámico

Historia:

Como administrador del sistema, quiero que las opciones del menú de reportes se carguen dinámicamente desde el backend según los permisos del usuario, para tener control centralizado de accesos.

Tareas Técnicas:
  1. Crear servicio /app/services/menuApi.ts con método fetchMenuOptions()
  2. Crear nuevo hook useMenuData() que maneje fetch y cache
  3. Modificar ReportsMenuPage para usar useMenuData en lugar de datos estáticos
  4. Actualizar types/menu.ts si API incluye nuevos campos
  5. Agregar loading skeleton mientras se cargan opciones
  6. Implementar error handling y retry logic
  7. Agregar tests para validar permisos
Estimación: 8 puntos

Ejemplo 3: Agregar Búsqueda en Menú

Historia:

Como usuario del sistema con muchas opciones en el menú, quiero poder buscar opciones por nombre, para encontrar rápidamente la funcionalidad que necesito sin tener que leer toda la lista.

Tareas Técnicas:
  1. Extender MenuScreen con prop opcional searchable: boolean
  2. Agregar TextField de búsqueda arriba de la lista de opciones
  3. Implementar lógica de filtrado por label y description
  4. Agregar highlight de texto coincidente
  5. Mantener numeración original de opciones (no renumerar filtradas)
  6. Agregar mensaje "No results found" cuando filtro no coincide
  7. Limpiar búsqueda con botón X
Estimación: 5 puntos

🧪 Patrones de Criterios de Aceptación

Para Funcionalidades de Visualización

  • El componente renderiza sin errores en consola
  • Todos los textos (título, subtítulo, labels) son visibles
  • Los estilos se aplican correctamente (colores, espaciado, fuentes)
  • El diseño es responsive en móvil, tablet, y desktop
  • No hay overlapping de elementos en ningún tamaño de pantalla
  • Accesibilidad: aria-labels presentes, contraste adecuado

Para Funcionalidades de Navegación

  • Click en opción válida navega a la ruta correcta
  • Input numérico + Enter navega a la opción correspondiente
  • Opciones disabled no permiten navegación
  • Loading state se muestra durante transición
  • URL cambia correctamente después de navegación
  • Historial del navegador se actualiza (botón back funciona)

Para Funcionalidades de Control de Acceso

  • Usuario no autenticado es redirigido a /login
  • Usuario sin permisos es redirigido a página apropiada
  • Opciones adminOnly solo visibles para usuarios admin
  • Badge "Admin" se muestra en opciones correspondientes
  • Verificación de rol ocurre en useEffect de la página
  • Redux state (isAuthenticated, user.role) se consulta correctamente

Para Funcionalidades de Manejo de Errores

  • Error se muestra en componente Alert con severity="error"
  • Mensaje de error es descriptivo y ayuda al usuario
  • Error no bloquea completamente la UI
  • Usuario puede intentar de nuevo después de error
  • Loading state se limpia después de error
  • Error se registra en consola para debugging
