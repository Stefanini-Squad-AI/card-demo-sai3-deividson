🔐 Seguridad y Autenticación

Guía de Desarrollo de User Stories - Sistema Legacy CardDemo COBOL

📋 Resumen del Módulo

ID del Módulo: security-auth

Propósito: Gestionar autenticación de usuarios, control de acceso basado en tipos de usuario y protección de transacciones CICS en el sistema mainframe CardDemo

Contexto de Negocio: Punto de entrada obligatorio al sistema. Todos los usuarios deben autenticarse antes de acceder a funcionalidades de gestión de cuentas, tarjetas y transacciones.

📦 Componentes Principales

  • COSGN00C (260 líneas) - Programa de sign-on/autenticación. Transacción: CC00
  • COUSR00C (695 líneas) - Listado paginado de usuarios. Transacción: CU00
  • COUSR01C (299 líneas) - Creación de nuevos usuarios Admin/Regular. Transacción: CU01
  • COUSR02C (414 líneas) - Eliminación de usuarios con confirmación. Transacción: CU02
  • COUSR03C (359 líneas) - Mantenimiento de seguridad y cambio de passwords. Transacción: CU03

📁 Archivos de Datos

  • USRSEC - VSAM KSDS (Key Sequenced Data Set)
    • Clave primaria: User ID (8 caracteres)
    • Longitud de registro: 80 bytes
    • Campos: User ID, First Name, Last Name, Password, User Type

🖥️ Pantallas BMS

  • COSGN00 - Pantalla de inicio de sesión (24x80)
  • COUSR00 - Lista de usuarios con paginación (10 por página)
  • COUSR01 - Formulario de creación/actualización de usuario
  • COUSR02 - Confirmación de eliminación de usuario
  • COUSR03 - Mantenimiento de seguridad y cambio de password

🔄 Patrones Reutilizables

Validación de Sesión CICS:

IF EIBCALEN = 0
   MOVE 'Session not found' TO WS-MESSAGE
   PERFORM SEND-ERROR-SCREEN
ELSE
   IF CDEMO-USER-TYPE NOT = 'A'
      MOVE 'Access denied - Admin only' TO WS-MESSAGE
      PERFORM SEND-ERROR-SCREEN
   END-IF
END-IF

Lectura VSAM con Manejo de Errores:

EXEC CICS READ FILE('USRSEC')
    INTO(SEC-USER-RECORD)
    RIDFLD(WS-USER-ID)
    RESP(WS-RESP-CD)
END-EXEC

EVALUATE WS-RESP-CD
    WHEN DFHRESP(NORMAL)
        PERFORM PROCESS-RECORD
    WHEN DFHRESP(NOTFND)
        MOVE 'User not found' TO WS-MESSAGE
    WHEN OTHER
        MOVE 'File error occurred' TO WS-MESSAGE
END-EVALUATE

🌐 APIs/Operaciones Públicas

Lectura de Usuario:

EXEC CICS READ FILE('USRSEC')
    INTO(SEC-USER-RECORD)
    RIDFLD(WS-USER-ID)
    RESP(WS-RESP-CD)
END-EXEC

Creación de Usuario:

EXEC CICS WRITE FILE('USRSEC')
    FROM(SEC-USER-DATA)
    RIDFLD(SEC-USR-ID)
    RESP(WS-RESP-CD)
END-EXEC

Actualización de Usuario:

EXEC CICS REWRITE FILE('USRSEC')
    FROM(SEC-USER-DATA)
    RESP(WS-RESP-CD)
END-EXEC

Eliminación de Usuario:

EXEC CICS DELETE FILE('USRSEC')
    RESP(WS-RESP-CD)
END-EXEC

📜 Reglas de Negocio

🔐 Autenticación

  • User ID: obligatorio, exactamente 8 caracteres alfanuméricos
  • Password: obligatorio, exactamente 8 caracteres
  • Ambos campos se convierten a mayúsculas automáticamente
  • Sin espacios en blanco permitidos
  • Tipos de usuario: A (Admin), U (User), G (Guest)

🛡️ Seguridad de Sesión

  • Sesión se establece con COMMAREA después de login exitoso
  • COMMAREA contiene User ID, User Type y timestamp
  • Timeout de inactividad: 15 minutos (configuración CICS)
  • Máximo 3 intentos de login fallidos consecutivos
  • Después de 3 intentos, usuario queda bloqueado
  • Desbloqueo requiere intervención de Admin

👥 Gestión de Usuarios

  • Solo usuarios Admin pueden crear/eliminar usuarios
  • User ID debe ser único en archivo USRSEC
  • No se puede eliminar el usuario en sesión actual
  • No se puede eliminar el último usuario Admin del sistema
  • Eliminación requiere confirmación explícita (S/N)
  • Cambio de password requiere conocer password anterior (excepto Admin)

📋 Criterios de Aceptación Comunes

Para Login de Usuario:

  • ✅ Campos User ID y Password visibles y funcionales
  • ✅ Password no muestra caracteres ingresados (no-display)
  • ✅ Validación de campos completos antes de procesar
  • ✅ Verificación de existencia de usuario en USRSEC
  • ✅ Comparación de password ingresado vs almacenado
  • ✅ Establecimiento de COMMAREA con datos de sesión
  • ✅ Redirección a menú principal (CMEN) después de login exitoso
  • ✅ Mensaje de error claro si credenciales inválidas
  • ✅ Bloqueo de usuario después de 3 intentos fallidos

Para Gestión de Usuarios:

  • ✅ Solo usuarios Admin pueden acceder a transacciones CU00-CU03
  • ✅ Validación de User Type en cada operación restringida
  • ✅ Mensajes de confirmación/error descriptivos en español
  • ✅ Prevención de operaciones no autorizadas (ej: eliminar usuario actual)
  • ✅ Confirmación explícita requerida para operaciones destructivas
